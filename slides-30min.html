BTRFS, Agents, and the
Post-Container Stack

Two ideas, one primitive:

  1. Replicate LLM models with filesystem-level deltas
  2. Deploy AI agents as Linux users, not containers

github.com/davdunc/btrfs-replication-test

The Problem & The Insight

  • 70B parameter model at fp16 ≈ 140 GB
  • Fine-tuning changes a fraction of the weights
  • Redeploying to N servers = N × 140 GB transferred
btrfs send -p <parent-snapshot> <child-snapshot>
  • Copy-on-write tracks which blocks actually changed
  • Send stream = only modified extents, no checksumming
  • Receive applies delta atomically on target

One-to-many via tee — single read, parallel delivery.
Snapshots are O(1). Rollback is instant.

Demo: Full + Incremental Send

# Snapshot and full send
btrfs subvolume snapshot -r /data/model /data/model-snap-v1
btrfs send --proto 2 --compressed-data /data/model-snap-v1 | \
  ssh fedora@target 'sudo btrfs receive /data/'

# After fine-tuning: incremental send (delta only)
btrfs subvolume snapshot -r /data/model /data/model-snap-v2
btrfs send --proto 2 --compressed-data \
  -p /data/model-snap-v1 /data/model-snap-v2 | \
  ssh fedora@target 'sudo btrfs receive /data/'

--proto 2 --compressed-data: v2 stream, no decompression overhead (kernel 6.0+)
⚠️ Never flip received snapshots to read-write — breaks received_uuid

The Elephant in the Room

CoW tracks writes, not value changes

  torch.save(model, "weights.bin")     ← writes ALL blocks → ALL are new
  btrfs send -p snap-v1 snap-v2       ← sees all-new blocks → sends 140GB

  mmap'd tensor file, update layer 31  ← writes 2% of blocks → 2% are new
  btrfs send -p snap-v1 snap-v2       ← sees 2% new blocks → sends ~2.8GB
Checkpoint patternIncremental send
torch.save() / safetensors writeFull model (all blocks new)
save_pretrained() to new fileFull model
mmap + partial write (in-place)Only touched blocks
Sharded — unchanged shardsZero bytes
Agent session memory (append)Only new blocks

Standard ML checkpointing defeats the delta. This is a format problem.

Making the Delta Real

btrfs as transport primitive + CoW-friendly format

  • Flat mmap'd tensor files with stable offsets per layer
  • Fine-tuning writes only to modified layer pages
  • LoRA adapters: base weights never written → zero delta
  • GGUF already supports mmap for inference — extend to checkpointing

The interesting research question:
Design a checkpoint format where btrfs send -p = semantic diff.
btrfs is the transport. The format makes it efficient.

TRANSITION

btrfs as a replication primitive for model weights.

Now: what if we use the same primitive for everything —
agent state, memory, work products?

And what if the agents themselves are just Linux users?

Agents as Service Account Users

Containers solved packaging. Agents don't have a packaging problem.

  • An agent is: runtime + model + memory + tools
  • A Linux user provides: UID/GID, cgroups, home dir, credential scope
  • K8s orchestrates where things run. Agents decide what to do.
2006: EC2 user-data     — shell scripts
2010: cloud-init        — declarative config
2015: Dockerfiles       — container packaging
2018: Helm charts       — container orchestration
2026: cloud-init agents — agent provisioning  ← full circle

Why /etc/agent-config/ instead of ~/.config/
Agents cannot grant themselves new capabilities. MCP config is system policy.
