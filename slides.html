BTRFS, Agents, and the
Post-Container Stack

Two ideas, one primitive:

  1. Replicate LLM models with filesystem-level deltas
  2. Deploy AI agents as Linux users, not containers

github.com/davdunc/btrfs-replication-test

The Problem: Model Distribution

  • 70B parameter model at fp16 ≈ 140 GB
  • Fine-tuning changes a fraction of the weights
  • Redeploying to N servers = N × 140 GB transferred
  • S3 sync, rsync, container layers — all file-level granularity

Every training iteration triggers full redistribution.

The Insight: Filesystem-Level Deltas

btrfs send -p <parent-snapshot> <child-snapshot>
  • Copy-on-write tracks which blocks actually changed
  • Send stream = only modified extents
  • No checksumming, no file-level diffing
  • Receive applies delta atomically on target

For 140GB model with 2GB changed → transmit ~2GB
rsync reads all 140GB just to figure that out.

Demo Environment

Fedora Cloud Base 43 on EC2 — two instances, btrfs data volumes

# User data: runs at boot
dnf install -y btrfs-progs

mkfs.btrfs -f -L btrfs-data /dev/nvme1n1
mkdir -p /data
mount -o noatime /dev/nvme1n1 /data

# Source only: create the model subvolume
btrfs subvolume create /data/model
  • AMI: Fedora-Cloud-Base-AmazonEC2.x86_64-43
  • Separate user-data for source and target
  • 50 GB gp3 volumes formatted as btrfs

Demo: Write, Snapshot, Full Send

# Write model data
dd if=/dev/urandom of=/data/model/weights.bin bs=1M count=100
echo '{"model":"test-llm","version":"1.0"}' > /data/model/config.json

# Read-only snapshot (required for send)
btrfs subvolume snapshot -r /data/model /data/model-snap-v1

# Full send to target (protocol v2, compressed)
btrfs send --proto 2 --compressed-data /data/model-snap-v1 | \
  ssh fedora@target 'sudo btrfs receive /data/'
  • Snapshot is instantaneous — CoW metadata operation
  • --proto 2 --compressed-data: no decompression overhead

Demo: Incremental Send

# Model updated by training
dd if=/dev/urandom of=/data/model/weights.bin bs=1M count=100
echo '{"model":"test-llm","version":"2.0"}' > /data/model/config.json

# New snapshot
btrfs subvolume snapshot -r /data/model /data/model-snap-v2

# Incremental: only the delta between v1 and v2
btrfs send --proto 2 --compressed-data \
  -p /data/model-snap-v1 /data/model-snap-v2 | \
  ssh fedora@target 'sudo btrfs receive /data/'

⚠️ Never flip received read-only snapshots to read-write
— breaks received_uuid. Create a new writable snapshot instead.

One-to-Many & Performance

btrfs send -p /data/model-snap-v1 /data/model-snap-v2 | \
  tee >(ssh fedora@target2 'sudo btrfs receive /data/') \
      >(ssh fedora@target3 'sudo btrfs receive /data/') | \
  ssh fedora@target4 'sudo btrfs receive /data/'

Comparison with Alternatives

btrfs wins on incremental (no checksumming) and rollback (local snapshots).
Tradeoff: need btrfs on both ends.

TRANSITION

btrfs as a replication primitive for model weights.

Now: what if we use the same primitive for everything —
agent state, memory, work products?

And what if the agents themselves are just Linux users?

Agents as Service Account Users

Containers solved packaging. Agents don't have a packaging problem.

  • An agent is: runtime + model + memory + tools
  • A Linux user provides: UID/GID, cgroups, home dir, credential scope
  • K8s orchestrates where things run. Agents decide what to do.
  • Orchestrating decision-makers with a decision-making system is redundant.
2006: EC2 user-data     — shell scripts
2010: cloud-init        — declarative config
2015: Dockerfiles       — container packaging
2018: Helm charts       — container orchestration
2026: cloud-init agents — agent provisioning  ← full circle

cloud-init: The Agent Provisioner

#cloud-config
users:
  - name: agent-build
    system: true
    shell: /usr/bin/agent-shell
    groups: [agents]
    no_create_home: true

runcmd:
  - btrfs subvolume create /data/agents/agent-build
  - mkdir -p /data/agents/agent-build/{memory,tools,work}
  - chown -R agent-build:agents /data/agents/agent-build
  - mount --bind /data/agents/agent-build /home/agent-build

  # MCP config — root-owned, agent-readable
  - install -m 0644 -o root -g agents \
      /dev/stdin /etc/agent-config/agent-build/mcp.json

  - systemctl enable --now agent@build
// /etc/agent-config/agent-build/mcp.json
{ "mcpServers": {
    "filesystem": { "command": "mcp-fs", "args": ["--root", "/home/agent-build"] },
    "btrfs": { "command": "mcp-btrfs", "args": ["--subvolume", "/data/agents/agent-build"] }
  },
  "permissions": {
    "allowedPaths": ["/home/agent-build", "/data/model"],
    "deniedPaths": ["/etc", "/root", "/home/agent-*"],
    "allowedCommands": ["btrfs subvolume snapshot", "btrfs send"]
  }
}

systemd: Lifecycle Without an Orchestrator

# /etc/systemd/system/agent@.service
[Service]
Type=notify
User=root
ExecStart=/usr/libexec/agent-monitor \
  --config /etc/agent-config/agent-%i/mcp.json \
  --user agent-%i --work-dir /home/agent-%i
Restart=always

# Resource isolation — the "container" is a cgroup
CPUQuota=200%
MemoryMax=8G

# Security — the "sandbox" is systemd
ProtectSystem=strict
ProtectHome=tmpfs
BindReadOnlyPaths=/etc/agent-config/agent-%i
BindPaths=/home/agent-%i
PrivateTmp=true
NoNewPrivileges=true
  • BindReadOnlyPaths for MCP config — read, not modify
  • agent-monitor is the privileged parent (next slide)

Privilege Separation & MCP Config

The OpenSSH model, applied to agents

 OpenSSH privsep                    Agent privsep
 ──────────────                     ─────────────
 sshd (root)                        agent-monitor (root)
   reads sshd_config                  reads /etc/agent-config/mcp.json
   └─ sshd (sshd user)               └─ agent-runtime (agent UID)
       └─ shell (user)                   └─ MCP tools (sandboxed)

agent-runtime (agent-build)              agent-monitor (root)
        │                                         │
        ├── "snapshot /data/model" ──────────────▶ │
        │                                  check mcp.json:
        │                                  allowedCommands ✅
        │                                  ◀── execute + return ──┤
        │                                         │
        ├── "read /etc/shadow" ──────────────────▶ │
        │                                  check mcp.json:
        │                                  deniedPaths ❌
        │                                  ◀── DENIED ───────────┤

Why /etc/agent-config/ instead of ~/.config/
Agents cannot grant themselves new capabilities.
MCP config is system policy, not user preference — same as sshd_config.

btrfs: Universal Replication Layer

One primitive for everything

# Replicate model weights (Part 1)
btrfs send --proto 2 --compressed-data \
  -p /data/model-snap-v1 /data/model-snap-v2 | \
  ssh target 'sudo btrfs receive /data/'

# Replicate agent session memory (Part 2)
btrfs send --proto 2 --compressed-data \
  -p .../memory/session-snap-prev .../memory/session-snap-curr | \
  ssh target 'sudo btrfs receive /data/agents/agent-build/memory/'

# Migrate entire agent to new host
btrfs send /data/agents/agent-build-snap | \
  ssh new-host 'sudo btrfs receive /data/agents/'

Models, memory, work products, full agent migration.
Same mechanism. No registry. No image layers.
